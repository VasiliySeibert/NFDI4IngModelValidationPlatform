## Files Involved

### `scripts/merge_docs_to_notebook.py`

Python script that merges a markdown doc into a notebook. No external dependencies (stdlib only).

**Key behaviors:**
- Reads the markdown file and the `.ipynb` file
- Replaces relative image paths (`![alt](./foo.svg)`) with raw GitHub URLs (`https://raw.githubusercontent.com/REPO/BRANCH/path/foo.svg`)
- Builds a Binder badge URL from `--repo` and `--branch` arguments
- Creates a markdown cell with an `<!-- AUTOGENERATED-DOCS -->` marker at the top, followed by the Binder badge, followed by the full doc content
- If the notebook already has a cell starting with this marker, it replaces it (idempotent). Otherwise inserts at position 0.

**CLI arguments:**
- `--doc` (required): path to markdown file
- `--notebook` (required): path to `.ipynb` file
- `--repo` (default: `VasiliySeibert/NFDI4IngModelValidationPlatform`): GitHub owner/repo
- `--branch` (default: `main`): git ref used in Binder URL and raw GitHub image URLs

### `.github/workflows/merge-docs-to-notebooks.yml`

GitHub Actions workflow.

**Triggers:**
- Push to `main` when files change in `docs/benchmarks/**`, `notebooks/**`, or `scripts/merge_docs_to_notebook.py`
- Manual dispatch (`workflow_dispatch`)

**Steps:**
1. Checkout repo
2. Set up Python 3.12
3. Run merge script with `--branch "binder-latest"`
4. If notebooks changed: commit with `[skip ci]`, push, then force-move the `binder-latest` tag to HEAD and push the tag

**Permissions:** `contents: write` (needed for push and tag operations)

The notebook (`plate_with_hole_fenics.ipynb`) is **not self-contained**. It requires:

1. **Simulation result files**: reads `summary_*.json` from `benchmarks/linear-elasticity-plate-with-hole/` and associated VTU/VTK/PVTU mesh output files
2. **Local package `meshhelper`**: imported from `src/meshhelper/` â€” provides `pyvista_mesh_to_dolfinx` conversion utilities
3. **Heavy Python dependencies**: `dolfinx`, `pyvista`, `meshio`, `mpi4py`, `ufl`, `pint`, `numpy`, `matplotlib`
4. **Environment file**: `environment.yml` defines the conda environment Binder uses to build

## The `binder-latest` Tag Strategy

### Problem solved
Binder caches Docker images keyed by git ref. When pointing to `main`, Binder may serve a stale cached image with old notebook content. Pinning to a commit SHA was tried but failed because:
- Chicken-and-egg: the SHA isn't known until after committing, but the URL must be in the notebook before committing
- A two-commit approach (commit, get SHA, update URL, commit again) left the first commit's SHA unreachable/orphaned, causing Binder errors like `Could not resolve ref ... Could not parse object '<sha>'`

### Current solution
- Use a **git tag** `binder-latest` as the ref in the Binder URL
- The tag name is stable and known ahead of time (no chicken-and-egg problem)
- After each pipeline run, the tag is force-moved to the latest commit
- Binder rebuilds when the tag points to a new commit

### Trade-off
Each time the tag moves, Binder needs to rebuild the Docker image (~1-3 min cold start). This is unavoidable when the environment or repo content changes.

## Known Issues / Open Items

1. **Tag must exist on remote**: The first pipeline run creates the tag. If it doesn't exist yet, run manually:
   ```bash
   git tag binder-latest HEAD
   git push origin binder-latest
   ```

2. **Binder cold starts**: Since each tag update points to a new commit, Binder rebuilds from scratch. If this becomes a problem, consider switching to `nbgitpuller` (Binder builds the environment once, pulls fresh notebook files at launch time).

3. **Only one notebook/doc pair is configured**: Adding more pairs requires adding more `python scripts/merge_docs_to_notebook.py` calls to the workflow. The script itself is generic and handles any pair.

4. **Image paths in docs**: The script converts relative image paths to `raw.githubusercontent.com` URLs. This requires the repo to be **public** for images to render in Binder. The repo `VasiliySeibert/NFDI4IngModelValidationPlatform` must be public for both Binder and image rendering to work.

5. **Google Colab was considered and rejected**: Colab opens notebooks instantly from GitHub (no build step) but does not have `dolfinx` and other heavy FEM dependencies pre-installed. Since the notebook must be executable, Binder is the better choice as it builds from `environment.yml`.

## Design Decisions Log

| Decision | Alternatives considered | Reason |
|---|---|---|
| Binder for execution | Google Colab, local Jupyter, JupyterHub | Colab lacks dolfinx; local Jupyter only works for one user; JupyterHub requires infrastructure |
| `binder-latest` tag | Commit SHA pinning, pointing to `main` | SHA pinning caused orphan-commit errors; `main` serves stale cached notebooks |
| `[skip ci]` on auto-commits | No skip | Prevents infinite loop: push triggers pipeline triggers push |
| `<!-- AUTOGENERATED-DOCS -->` marker | Separate metadata field, cell tags | Simple, visible, works with plain JSON parsing |
| Raw GitHub URLs for images | Base64 embedding, relative paths | Simplest; requires public repo but avoids notebook bloat |
